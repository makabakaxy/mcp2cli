# Skill File Generation Rules

## Goal

From a CLI mapping file (`cli/<server>.yaml`) and tool schema (`tools/<server>.json`),
generate a Skill file collection usable by agents, enabling agents to efficiently use mcp2cli commands
while reducing context consumption from ~5000 tokens (raw MCP tool schema) to ~600 tokens.

## Output File Structure

### SKILL.md (Main File)

Automatically loaded into every agent conversation; must be extremely concise.

**Hard limit: <= 600 tokens**

**Body structure (in this order):**

1. Frontmatter (name + description + source_version + source_cli_hash + generated_at)
2. `# <server> (via mcp2cli)` + one-line summary
3. `## Shortcuts` — shortcut list (no need to explain equivalent forms)
4. `## Commands` — command overview table (split by group with ###, including Example and Ref columns)
5. `## Discover Parameters` — point to --help and reference/
6. `## User Notes` — fixed MUST READ link to users/SKILL.md (generated by LLM)

**Command form priority (shortest form first):**

1. Has `command_shortcuts` → use shortcut: `mcp2cli jira issue create`
2. Has `server_aliases` → use alias: `mcp2cli atlassian jira issue create`
3. Neither → use full name: `mcp2cli mcp-atlassian jira issue create`

**Command selection (max 8 per group):**

Priority: create > get > search > list > update > delete > others
Sub-resource operations (comment, watcher, worklog, label, attachment, etc.) are not included in the main file.

**Commands table column descriptions:**

| Column | Description |
|---|---|
| Command | Command path (using shortest form) |
| Description | Action description |
| Example | Common examples (complete commands), multiple separated by `<br>`. High-frequency commands 2-3 examples, low-frequency commands 1 or empty |
| Ref | Link to detailed file under reference/ |

> **Note**: Ref column links point to files under the reference/ directory, containing full parameter descriptions and more usage examples for that command. Agents should read the corresponding reference file when more information is needed.

### reference/<group>.md or reference/<group>-<resource>.md

**Split rules:**
- Group has <= 5 leaf commands → merge into `<group>.md`
- Group has > 5 leaf commands → split by resource

**Content for each command:**
- 1-3 usage examples (showing required parameters)
- "Also supports" line listing optional parameter names
- Parameter names extracted from inputSchema, converted to kebab-case

**Parameter name conversion:**
- snake_case → kebab-case: `project_key` → `--project-key`
- camelCase → kebab-case: `issueKey` → `--issue-key`

**Single file limit: <= 200 lines**

### users/ Directory

User-customizable content area, never overwritten by generate/update. On first generation, create the `users/` directory, a `.gitkeep` placeholder file, and `workflows.md`.

**users/workflows.md**:
- On first generation, the LLM writes 5-10 common multi-step workflow scenarios
- Organized by workflow (e.g., "JIRA Workflow", "Confluence Workflow")
- Each scenario includes combined operations with multiple commands
- Use shortest command form
- **Never overwritten by generate/update** — users can freely edit and extend

> **Note**: workflows.md complements the Example column in the Commands table. The Example column shows single-command usage, while workflows.md shows complex workflows requiring multiple commands working together (e.g., "create issue → assign → add to sprint → transition status").

The `## User Notes` section is generated by the LLM, with a fixed format:

```markdown
## User Notes

> **MUST READ** [users/SKILL.md](users/SKILL.md) for custom workflows and tips.
> See [users/workflows.md](users/workflows.md) for multi-step workflow examples.
> Not overwritten by updates.
```

## Frontmatter Specification

```yaml
---
name: <server-name>
description: <one-line English description listing core capability keywords>
source_version: "<version field from cli/*.yaml>"
source_cli_hash: "<first 8 hex characters of SHA-256 of cli/*.yaml content>"
generated_at: "<ISO 8601 timestamp>"
---
```

- `source_version`: From the CLI YAML's `version` field (i.e., MCP server version), for quick human version identification
- `source_cli_hash`: First 8 characters of the SHA-256 hash of the CLI YAML file content, computed by the program and passed into the prompt; the LLM writes it directly. Used by the program to determine whether the skill is in sync with the CLI
- `generated_at`: Generation or update timestamp

Description examples:
- `Manage JIRA issues, sprints, boards, and Confluence pages via CLI. Use when user needs to create/search/update JIRA tickets, manage sprints, or edit Confluence pages.`
- `Automate GitHub operations via CLI. Use when user needs to manage repos, pull requests, issues, or releases.`

## Description Writing Guidelines

- Language: English
- Format: `<what it does>. Use when <trigger conditions>.`
- Include triggerable keywords (verbs and nouns)
