# mcp2cli 设计总览

## 一、项目定位

mcp2cli 将 MCP (Model Context Protocol) server 的扁平 tool 列表，经过三级转换，最终产出 agent 高效使用的 Skill 文件：

```
MCP Server (扁平 tool 列表)
    │
    │  scan + AI 分析
    ▼
层级式 CLI 命令树 (渐进式披露，每层 --help 只展示下一层)
    │
    │  AI 生成
    ▼
Skill 文件 (≤600 tokens，agent 每次对话自动加载)
```

**三级产物各自的作用：**

| 产物 | 文件 | 作用 |
|------|------|------|
| tool 列表 | `tools/<server>.json` | MCP server 所有 tool 的完整 schema 快照 |
| **层级式 CLI** | `cli/<server>.yaml` | 将扁平 tool 组织为人类直觉的层级命令树，支持渐进式披露（`--help` 逐层展开） |
| Skill 文件 | `skills/<server>/SKILL.md` | 基于 CLI 命令树生成的精简摘要（≤600 tokens），agent 加载后通过 CLI 调用 tool |

用户通过两个核心命令一键完成全流程：

- **`mcp2cli convert`**：将已配置在 Claude/Cursor/Codex 中的 MCP server 一键转换，并在原客户端中关闭 MCP server
- **`mcp2cli install`**：安装并转换一个全新的 MCP server（AI 搜索安装信息 → 全流程自动化）

**核心价值：**

- **渐进式披露**：层级式 CLI 命令树让 agent/用户逐层探索（server → group → resource → action → params），每层 `--help` 只暴露下一层，避免一次性面对几十个 tool
- **Token 压缩 75%+**：将 MCP tool schema（通常 4000-6000 tokens）经 CLI 命令树压缩为 Skill 文件（约 600 tokens）
- **CLI 即文档**：命令树本身就是结构化文档，agent 通过 `--help` 按需获取参数详情，无需全量加载
- 兼容已有的大量 MCP 工具生态，无需改造 MCP server

## 二、总体架构

```
┌──────────────── 配置源 ─────────────────┐
│  ~/.claude.json                         │
│  ~/.cursor/mcp.json                     │
│  ~/.codex/config.toml                   │
│  ~/.agents/mcp2cli/servers.yaml         │
└──────────┬─────────────────┬────────────┘
           │                 │
   mcp2cli convert    mcp2cli install
   (已有配置)         (全新server)
           │                 │
           └────────┬────────┘
                    │ 内部自动执行三级转换
                    ▼
  ┌─ Stage 1: MCP Server → Tool 列表 ──────────┐
  │  mcp2cli scan                               │
  │  连接 server → list_tools → 写入             │
  │  tools/<server>.json (完整 schema 快照)      │
  └─────────────────┬──────────────────────────-┘
                    ▼
  ┌─ Stage 2: Tool 列表 → 层级式 CLI ──────────┐
  │  mcp2cli generate cli  (AI 分析)            │
  │  扁平 tool 名 → 层级命令树                   │
  │  cli/<server>.yaml                          │
  │                                             │
  │  例: jira_create_issue                      │
  │    → mcp2cli jira issue create              │
  │  渐进式披露: --help 每层只展示下一层          │
  └─────────────────┬───────────────────────────┘
                    ▼
  ┌─ Stage 3: CLI → Skill 文件 ────────────────┐
  │  mcp2cli generate skill  (AI 生成)          │
  │  CLI 命令树 → 精简摘要 (≤600 tokens)        │
  │  skills/<server>/SKILL.md + reference/      │
  │                                             │
  │  mcp2cli skill sync                         │
  │  软链接到 ~/.claude/skills/ 等               │
  └─────────────────┬───────────────────────────┘
                    ▼
  ┌─ 最终: 在客户端中关闭原 MCP server ────────┐
  │  (仅 convert) 设置 disabled: true           │
  └─────────────────────────────────────────────┘

  使用时:
  ┌─────────────────────────────────────────────┐
  │  Agent 读取 SKILL.md (600 tokens)           │
  │  → 通过层级式 CLI 调用 tool                  │
  │  → CLI 经 Daemon 转发到 MCP Server          │
  │                                             │
  │  ┌─ MCP Proxy Daemon ────────────────────┐  │
  │  │  Unix Socket, 连接池, 两层空闲回收    │  │
  │  └──────────────────────────────────────-┘  │
  └─────────────────────────────────────────────┘
```

## 三、CLI 命令体系

### 3.1 命令层级

| 层级 | 类型 | 命令形式 |
|------|------|----------|
| 核心命令 | 一键转换/移除 | `mcp2cli convert <server>` — 已有配置转 skill<br>`mcp2cli install <server>` — 安装新 server 并转 skill<br>`mcp2cli remove <server>` — 移除 server 及所有产物 |
| 管理命令 | 单步操作 | `mcp2cli list / scan / generate / validate / daemon` |
| 直通命令 | 底层直通 | `mcp2cli tools <server> [tool]`<br>`mcp2cli call <server> <tool> [--args]` |
| 层级命令 | AI 生成 | `mcp2cli <server> <group> <sub> <action>` |

### 3.2 核心命令（一键转换）

```bash
# 将已配置在 Claude/Cursor/Codex 中的 MCP server 转为 skill
# 自动执行：提取配置 → scan → generate cli → generate skill → skill sync → disable 原配置
mcp2cli convert mcp-atlassian
mcp2cli convert mcp-atlassian --skip-disable   # 不关闭原mcp server配置, skill和mcp server同时存在
mcp2cli convert mcp-atlassian --source claude  # 指定从哪个客户端读取配置

# 安装并转换一个全新的 MCP server（系统中尚未配置的）
# 自动执行：AI 搜索安装信息 → 交互补全 env → scan → generate → skill sync
mcp2cli install mcp-atlassian
mcp2cli install mcp-atlassian --env JIRA_URL=https://... --yes
```

**两个命令的区别：**

| 维度 | `convert`（已有 server） | `install`（新 server） |
|------|--------------------------|------------------------|
| 适用场景 | 已在 Claude/Cursor/Codex 中配置好 | 系统中尚未配置 |
| 配置来源 | 从客户端配置文件提取 | AI 搜索互联网 |
| 环境变量 | 从已有配置提取（无需交互） | 交互式提示用户输入 |
| 关闭原配置 | 设置 `disabled: true` | 无（没有原配置） |

详细设计：[12-convert.md](12-convert.md)、[11-install.md](11-install.md)

### 3.2.1 移除命令（install/convert 的反向操作）

```bash
# 移除 server 及所有产物（symlink、skill、cli 映射、tools 缓存、servers.yaml 配置）
# convert 过的 server 会自动在客户端配置中重新启用（撤销 disabled: true）
mcp2cli remove mcp-atlassian
mcp2cli uninstall mcp-atlassian              # 别名

# 选项
mcp2cli remove mcp-atlassian --keep-config   # 保留 servers.yaml 配置，仅清理生成文件
mcp2cli remove mcp-atlassian --purge-package # 同时卸载底层 package
mcp2cli remove mcp-atlassian --dry-run       # 预览将执行的操作
mcp2cli remove mcp-atlassian --force         # 跳过确认提示

# 仅从 servers.yaml 移除配置
mcp2cli mcp remove mcp-atlassian

# 仅移除客户端 skill 目录的 symlink
mcp2cli skill unsync mcp-atlassian
```

**remove 与 install/convert 的对应关系：**

| install/convert 步骤 | remove 反向操作 |
|------|------|
| skill sync（创建 symlink） | 删除各客户端 skill symlink |
| generate skill（生成 SKILL.md） | 删除 `skills/<server>/` 目录 |
| generate cli（生成命令树） | 删除 `cli/<server>.yaml` |
| scan（扫描 tool 列表） | 删除 `tools/<server>.json` |
| mcp install / convert 写入 servers.yaml | 从 servers.yaml 移除条目 |
| convert 的 disable 原配置 | 重新启用客户端配置（移除 disabled 字段） |

详细设计：[13-uninstall.md](13-uninstall.md)

### 3.3 管理命令（单步操作）

```bash
# 列出所有配置源中的 MCP server
mcp2cli list

# 扫描 server 的 tool 列表，写入 tools/<server>.json，返回文件路径
mcp2cli scan mcp-atlassian
mcp2cli scan --all

# AI 生成层级命令映射文件
mcp2cli generate cli mcp-atlassian

# 生成 SKILL.md（基于层级命令树）
mcp2cli generate skill mcp-atlassian [-o ~/.agents/mcp2cli/skills/mcp-atlassian/]

# 将 skill 文件软链接到各 AI 客户端
mcp2cli skill sync mcp-atlassian

# 校验命令映射文件
mcp2cli validate mcp-atlassian

# Daemon 管理
mcp2cli daemon status
mcp2cli daemon stop
```

### 3.4 底层直通命令

直接使用 MCP tool 原始名称，不经过层级映射。始终可用，不依赖 `generate cli`：

```bash
# 列出所有 tool
mcp2cli tools mcp-atlassian

# 查看某个 tool 的参数
mcp2cli tools mcp-atlassian jira_create_issue

# 直接调用（使用 MCP tool 原始名称）
mcp2cli call mcp-atlassian jira_create_issue --project-key INFRA --summary "Fix bug"
```

经过 `mcp2cli generate cli` 后可用。渐进式披露——每层 `--help` 只暴露下一层的信息：

## 四、CLI 映射文件设计

`mcp2cli generate cli` 的产物，存储在 `~/.agents/mcp2cli/cli/<server>.yaml`。

### 4.1 tools JSON 文件格式

`mcp2cli scan` 的产物，存储在 `~/.agents/mcp2cli/tools/<server>.json`：

```json
{
  "server": "mcp-atlassian",
  "version": "1.2.3",
  "scanned_at": "2026-04-02T10:00:00Z",
  "tools": [
    {
      "name": "jira_create_issue",
      "description": "Create a new JIRA issue",
      "inputSchema": { "..." : "..." }
    }
  ]
}
```

- `version`：来自 MCP server handshake 时返回的 `serverInfo.version`，若 server 未提供则为 `null`
- 命令解析时读取 `tools` 数组校验参数合法性；`version` 用于与 CLI YAML 中的 `version` 比对，检测 server 是否已升级

### 4.2 CLI YAML 文件格式

```yaml
# ~/.agents/mcp2cli/cli/mcp-atlassian.yaml
server: mcp-atlassian
version: "1.2.3"          # 来自扫描时的 tools JSON，用于检测 server 升级
generated_at: "2026-04-02T10:00:00Z"
generated_by: ai          # "ai" 或 "manual"

# server 名称别名：替代第一个子命令
server_aliases:
  - atlassian              # mcp2cli atlassian jira issue → mcp2cli mcp-atlassian jira issue

# 命令快捷方式：跳过 server token，直接匹配子命令
command_shortcuts:
  - jira                   # mcp2cli jira issue create → mcp2cli mcp-atlassian jira issue create
  - confluence             # mcp2cli confluence page get → mcp2cli mcp-atlassian confluence page get

commands:
  jira:
    _description: "JIRA project management"
    issue:
      _description: "Issue operations"
      create:
        _tool: jira_create_issue
        _description: "Create a new JIRA issue"
        _examples:
          - "mcp2cli mcp-atlassian jira issue create --project-key INFRA --summary 'Fix bug'"
      get:
        _tool: jira_get_issue
        _description: "Get issue details by key"
      search:
        _tool: jira_search
        _description: "Search issues using JQL"
      update:
        _tool: jira_update_issue
        _description: "Update an existing issue"
    sprint:
      _description: "Sprint operations"
      create:
        _tool: jira_create_sprint
        _description: "Create a new sprint"
      list:
        _tool: jira_get_sprints_from_board
        _description: "List sprints for a board"
  confluence:
    _description: "Confluence wiki operations"
    page:
      _description: "Page operations"
      get:
        _tool: confluence_get_page
        _description: "Get page content"
      create:
        _tool: confluence_create_page
        _description: "Create a new page"
      update:
        _tool: confluence_update_page
        _description: "Update page content"
    search:
      _tool: confluence_search
      _description: "Search Confluence content"
```

**设计说明：**

- `_` 前缀字段为元数据，非 `_` 前缀为子命令节点
- 叶子节点包含 `_tool` 字段，指向 MCP tool 原始名称
- 中间节点只有 `_description`，用于 `--help` 展示
- `_param_aliases`：（预留，MVP 暂不实现）参数缩写别名
- `_examples` 在 `--help` 和 SKILL.md 中展示
- `server_aliases` 定义 server 名称的简短别名
- `command_shortcuts` 定义可跳过 server 直接使用的顶层命令
- 文件可由 AI 生成后人工微调

### 4.3 命令解析流程

```
输入: mcp2cli mcp-atlassian jira issue create --project-key INFRA
解析过程:
  1. "mcp-atlassian" → 在已生成的 cli/*.yaml 中查找
     → 找到 ~/.agents/mcp2cli/cli/mcp-atlassian.yaml
  2. "jira" → commands.jira (中间节点, 继续)
  3. "issue" → commands.jira.issue (中间节点, 继续)
  4. "create" → commands.jira.issue.create
     → 发现 _tool 字段 → 叶子节点, 停止路径解析
  5. 剩余参数 "--project-key INFRA" → 转为 tool 参数
     → 查 tools/mcp-atlassian.json 中 jira_create_issue 的 input_schema 校验参数合法性
  6. 转发到 daemon:
     → {"server": "mcp-atlassian", "tool": "jira_create_issue",
        "params": {"project_key": "INFRA"}}
```

### 4.4 找不到映射时的 Fallback

```
输入: mcp2cli mcp-atlassian some_unknown_tool --arg val
解析:
  1. "mcp-atlassian" → 找到 yaml
  2. "some_unknown_tool" → commands 中不存在
  3. Fallback: 当作 MCP tool 原始名称，等价于
     mcp2cli call mcp-atlassian some_unknown_tool --arg val
  4. 在 tools/mcp-atlassian.json 中查找 tool 是否存在
  5. 存在则直接调用，不存在则报错
```

这保证层级命令和直通命令在同一入口下兼容。

详细设计见 [4.3-fallback.md](4.3-fallback.md)。

### 4.5 别名与快捷路由

为了减少输入长度、提升使用效率，支持两种快捷路由方式：

#### 4.5.1 两种快捷方式

| 类型 | 配置字段 | 效果 |
|------|----------|------|
| Server 别名 | `server_aliases` | 替换第一个 token<br>`mcp2cli atlassian jira issue` → `mcp2cli mcp-atlassian jira issue` |
| 命令快捷方式 | `command_shortcuts` | 跳过 server token<br>`mcp2cli jira issue` → `mcp2cli mcp-atlassian jira issue` |

- `server_aliases`：值为字符串列表，每个元素是 server 名的简短替代。
- `command_shortcuts`：值为字符串列表，每个元素必须是 `commands` 下的顶层 key。

详细设计见 [4.4-alias-routing.md](4.4-alias-routing.md)（索引构建、解析算法、优先级规则、冲突检测、AI 生成规则、`--help` 展示、SKILL.md 别名展示）。

```

  │  - 写入 ~/.agents/mcp2cli/cli/mcp-atlassian.yaml │
```

### 5.3 AI 生成 vs 手动编辑

| 场景 | 方式 |
|------|------|
| 首次转换 | `mcp2cli generate cli` 让 AI 生成初始版本 |
| 微调分组 | 直接编辑 YAML 文件（改层级、改描述、加别名） |
| MCP server 新增 tool | `mcp2cli generate cli --merge`：AI 只处理新增的，保留已有调整 |
| 完全自定义 | 手写 YAML，`generated_by: manual` |

## 六、Skill 生成（基于层级命令树）

`mcp2cli generate skill` 读取 CLI 映射文件和 tool schema，通过 AI 生成 agent 可用的 Skill 文件集合。支持**增量更新**：当 skill 文件已存在时，自动对比 CLI YAML 的变化，仅修改受影响的部分。

```bash
mcp2cli generate skill mcp-atlassian          # 首次生成 / 增量更新（自动判断）
mcp2cli generate skill mcp-atlassian --full   # 强制全量重新生成
```

### 6.1 输出目录结构

生成文件存放在 `~/.agents/mcp2cli/skills/<server>/`，同时在 `~/.agents/skills/<server>` 创建软链接，方便 agent 框架统一发现：

```
~/.agents/mcp2cli/skills/<server>/    # 实际存储位置
├── SKILL.md                          # 主文件（≤600 tokens），agent 每次对话加载
├── reference/                        # 详细参数 + 高频示例（按需读取）
│   ├── jira-issue.md
│   ├── jira-sprint.md
│   └── confluence-page.md
└── examples/                         # 多步骤复杂场景（按需读取）
    └── examples.md

~/.agents/skills/<server>             # 软链接 → ~/.agents/mcp2cli/skills/<server>/
```

### 6.2 SKILL.md 设计原则

- **精简**：≤600 tokens，只含命令总览（含内联示例）、快捷方式
- **优先使用 `command_shortcuts`**：`mcp2cli jira issue create`（非全名）
- **参数细节在 reference/**：高频命令带示例 + 其余参数列名 + `--help` 提示

### 6.4 AI 生成流程

同 `generate cli`，通过 LLM 读取生成规则、示例模板、CLI YAML 和 tools JSON，生成 SKILL.md + reference/ + examples/。支持 `--resume` 会话续接重试。

**增量更新**：当 skill 文件已存在时，通过 SKILL.md frontmatter 中的 `source_cli_hash`（CLI YAML 的 SHA-256 前 8 位）判断是否需要更新。hash 一致则跳过，不一致则 LLM 读取已有 skill 文件，对比 CLI YAML 差异，仅修改受影响的部分（新增/删除命令行、更新 reference 段落）。

详细设计见 [6.0-skill-generation.md](6.0-skill-generation.md)（增量更新流程、Prompt 模板、校验流程）。

**Agent 使用模式对比：**

| 方式 | Token 消耗 |
|------|-----------|
| 传统 MCP（加载全部 tool schema） | 12 个 tool 的完整 schema ≈ 5000 tokens，每轮对话都带着 |
| mcp2cli + Skill | SKILL.md ≈ 600 tokens + 按需 reference ≈ 200 tokens/次<br>一般一次对话调用 2-3 个 tool ≈ 1200 tokens，**节省 75%** |

## 七、Daemon 设计

### 7.1 架构

```
┌─────────────────────────────────────────────────┐
│                MCP Proxy Daemon                 │
│                                                 │
│  监听: ~/.agents/mcp2cli/daemon.sock            │
│  PID:  ~/.agents/mcp2cli/daemon.pid             │
│                                                 │
│  ┌─ 连接池 ───────────────────────────────────┐ │
│  │  mcp-atlassian  [alive]  last: 10:05       │ │
│  │  playwright     [idle]   last: 09:30       │ │
│  └────────────────────────────────────────────┘ │
│                                                 │
│  看门狗: 每 30s 检查一次                         │
└─────────────────────────────────────────────────┘
```

### 7.2 环境变量处理

MCP server 配置中的 `env`（如 API Token、URL）由 Daemon 在启动子进程时透明注入，CLI 调用方和 LLM 完全不感知。详见 [7.2-env-handling.md](7.2-env-handling.md)。

### 7.3 两层空闲回收

```
┌──────────── 第一层：Server 连接回收 ──────────────┐
│                                                  │
│  每个 MCP server 连接独立跟踪 last_used 时间戳    │
│                                                  │
│  某个 server 空闲 > SERVER_IDLE (默认 10 min)     │
│  → 关闭该 server 的子进程                         │
│  → 释放该连接的内存                               │
│  → 其他 server 连接不受影响                       │
│                                                  │
│  例：playwright 用了一次后 10min 没用              │
│  → 关闭 playwright 进程                          │
│  → mcp-atlassian 连接保持                        │
│                                                  │
└──────────────────────────────────────────────────┘
                      │
                      ▼
┌──────────── 第二层：Daemon 自身回收 ──────────────┐
│                                                  │
│  条件：所有 server 连接已关闭                      │
│       AND 无新请求 > DAEMON_IDLE (默认 5 min)     │
│  → Daemon 进程退出                               │
│  → 清理 PID 文件和 Socket 文件                    │
│                                                  │
│  绝对兜底：                                       │
│  Daemon 运行时间 > MAX_LIFETIME (默认 24h)        │
│  → 无条件退出（防止僵尸进程）                      │
│                                                  │
└──────────────────────────────────────────────────┘
```

### 7.4 生命周期管理

```
首次 CLI 调用:
  main.py → 检查 daemon.pid → 不存在/进程已死
  → fork daemon 子进程 (detach, start_new_session)
  → daemon 创建 Unix Socket, 写 PID 文件
  → main.py 等待 socket 文件出现 (最多 5s)
  → 连接 socket, 发送请求

后续调用:
  main.py → 检查 daemon.pid → 进程存活
  → 直接连接 socket, 发送请求 (~50ms)

崩溃恢复:
  main.py → 检查 daemon.pid → 文件存在但进程已死
  → 清理 stale PID 文件和 socket 文件
  → 重新启动 daemon
```

## 十、关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 核心入口 | `convert`（已有）+ `install`（新装） | 两种场景互补，覆盖所有用户起点 |
| convert 关闭策略 | disable 而非 remove | 安全可逆，保留原配置供回退 |
| 层级命令 vs 扁平命令 | 两者共存，层级命令优先，扁平 fallback | 层级对 agent 更友好，但不能丢失直通能力 |
| 映射文件格式 | YAML，`_` 前缀元数据 | 人类可读可编辑，AI 易生成，`_` 前缀区分元数据和子命令 |
| 映射文件位置 | `~/.agents/mcp2cli/cli/<server>.yaml` | 每个 server 独立文件，互不干扰 |
| AI 生成 vs 手写 | AI 初始化 + 人工微调 | AI 处理 80% 的分组逻辑，人工处理边缘情况 |
| 新增 tool 处理 | `generate cli --merge` | 保留已有调整，只处理新增 |
| Daemon 退出 | 两层空闲回收 + 24h 兜底 | 精细控制资源释放，防僵尸 |
| 配置优先级 | 自定义 > Claude > Cursor > Codex | 用户自定义覆盖一切 |

## 十一、扩展能力

已规划但不在 MVP 中的功能：

- `mcp2cli remove` / `mcp2cli uninstall`：`install` 和 `convert` 的反向操作。全流程清理：移除 skill symlink → 删除 skill/cli/tools 生成文件 → 从 servers.yaml 移除配置 → 重新启用被 convert disable 的客户端配置。支持 `--keep-config`（保留 servers.yaml）、`--purge-package`（卸载底层 package）、`--dry-run`。子命令 `mcp2cli mcp remove` 仅移除 servers.yaml 配置，`mcp2cli skill unsync` 仅移除 symlink。详细设计见 [13-uninstall.md](13-uninstall.md)。
- 交互模式：`mcp2cli mcp-atlassian -i` 进入交互式 REPL，自动补全命令路径
- 管道支持：`echo '{"jql":"project=INFRA"}' | mcp2cli mcp-atlassian jira issue search --json`
- 多 server 编排：`mcp2cli pipe mcp-atlassian:jira:issue:get | mcp-atlassian`
